# 项目优化方向 — 面试加分项

> 当前状态：6 项核心功能全部完成，端到端验证通过。以下为面试展示加分优化方向。

---

## 建议顺序

```
密钥安全 → 自动化测试 → Docker → 多轮 Demo → 错误处理 → 数据看板
  必做       必做        必做      加分        加分        加分
```

---

## 1. 密钥安全化（约 15 分钟）

**问题**：`workspace/config.json` 中硬编码了 `apiKey: sk-f694d39b...`，面试官一眼看出安全问题。

**方案**：抽到 `.env` 文件 + 环境变量引用，展示代码和配置分离的安全意识。

**涉及文件**：
- `workspace/config.json` — 将敏感字段改为环境变量占位
- `workspace/.env.example` — 已存在，确认字段完整

**效果**：面试官问"怎么管理密钥" → 你有话说。

---

## 2. 自动化测试（约 1 小时）

**问题**：当前零测试，面试官会问"你怎么保证改动不出问题？"

**方案**：最少加 3 个测试：

| 测试 | 内容 | 不依赖 |
|------|------|--------|
| `tests/test_ticket_server.py` | MCP 5 个工具单元测试（create/query/list/update/feedback） | LLM |
| `tests/test_faq_search.py` | FAQ 知识库匹配覆盖率 | LLM |
| `tests/test_e2e_scenario.py` | 端到端对话脚本 | 需要 LLM |

**关键**：前两个可以不依赖 LLM，跑得贼快，面试时现场跑一下。

---

## 3. Docker 一键部署（约 30 分钟）

**问题**：当前需要手动配 Python 环境、装依赖。

**方案**：`workspace/` 下已有 `Dockerfile` 和 `docker-compose.yml`，确认可正常构建运行。

**验证**：
```bash
cd workspace
docker compose up --build
```

**效果**：面试中一句 `docker compose up` 启动整个系统。

---

## 4. 多轮对话演示脚本（约 30 分钟）

**问题**：当前验证是单轮问答，没展示 bot 的上下文保持能力。

**方案**：准备一个演示脚本，模拟完整的客服对话流程：

```
用户: 我的数据同步卡住了
Bot:  [查 FAQ，给出排查步骤]
用户: 试了，没用
Bot:  [升级 urgent 工单，告知 SLA 30min]
用户: 好的，现在恢复了
Bot:  [确认解决 + 要满意度评分]
用户: 5 分
Bot:  [submit_feedback + 感谢关闭]
```

**形式**：一个 `demo.sh` / `demo.py` 脚本，跑完即完成演示。

---

## 5. 错误处理展示（约 30 分钟）

**问题**：各环节出错后行为未覆盖。

**方案**：准备可演示的容错场景：

| 场景 | 当前行为 | 期望行为 |
|------|---------|---------|
| LLM API 挂了 | 报错堆栈 | 给用户友好提示"系统繁忙，请稍后重试" |
| MCP Server 启动失败 | gateway 可能崩溃 | 降级运行，FAQ 仍可用 |
| 飞书 WebSocket 断开 | 未知 | 自动重连（框架已支持，确认生效） |

**涉及文件**：`nanobot/agent/loop.py`、`nanobot/channels/manager.py`

---

## 6. 数据看板（约 1-2 小时）

**问题**：工单数据都在 SQLite 里，但面试中无法直观展示。

**方案**：写一个简单的 `python dashboard.py`，终端输出统计：

```
今日数据概览
  工单总数:     12
  FAQ 命中率:   78%
  满意度均分:   4.2 / 5
  低分工单(≤2): 1  (TK-xxx)

优先级分布
  urgent:  2
  high:    3
  medium:  5
  low:     2
```

不需要前端，终端打印即可。面试时跑一下比口说"做了工单系统"有说服力 10 倍。
