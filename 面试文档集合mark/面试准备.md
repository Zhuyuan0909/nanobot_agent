## 二、关键设计决策（STAR 法则）

### 决策 1：为什么不用 Flask Webhook，直接用 nanobot ChannelManager？

**背景**：参考教程用 Flask 做多平台消息路由，需要公网 IP + Nginx + Webhook 对接。

**可选方案**：
- A) Flask + Webhook，各平台消息推送到同一个 HTTP 端点
- B) 直接用框架内置的 ChannelManager

**选择 B 的理由**：
- 框架已为飞书/钉钉等 16 个平台实现了 WebSocket 长连接的原生 SDK 对接
- 无需公网 IP，无需 Nginx 配置
- 消息自动路由到统一 AgentLoop

**效果**：零网络配置，本地 `nanobot gateway` 一条命令同时接入飞书和钉钉。

---

### 决策 2：MCP Server 为什么用原始 JSON-RPC 而不是 MCP SDK？

**背景**：nanobot 用 `mcp` SDK 做客户端连接外部 MCP 工具，需要为工单系统写一个服务端。

**可选方案**：
- A) 使用 `mcp.server` SDK 类
- B) 原始 JSON-RPC 2.0 over stdin/stdout

**选择 B 的理由**：
- MCP Python SDK 的服务端 API 版本敏感（`mcp.__version__` 甚至不可用），频繁变动
- JSON-RPC 2.0 协议本身比 SDK 稳定得多
- 工单系统只需要 4 个工具（create/get/list/update），180 行即可实现全部路由逻辑
- 关键坑点：所有日志必须输出到 stderr，否则 stdout 被污染导致 JSON-RPC 通道断开

**代价**：不能使用 SDK 提供的高级特性（如 resources、prompts），但工单场景不需要。

---

### 决策 3：知识库为什么用 Markdown + grep 而不是向量数据库？

**可选方案**：
- A) 向量数据库 + Embedding（如 Chroma + text-embedding）
- B) Markdown 文件 + grep 关键词搜索

**选择 B 的理由**：
- FAQ 当前 4 个文件 280 行，百条以内，关键词搜索命中率足够
- 零额外依赖，Agent 内置 `grep` 工具直接可用
- Markdown 文件可读可维护，人工编辑友好

**转向 A 的条件**：FAQ 条目超过 500+，或需要语义匹配而非精确关键词匹配。

---

### 决策 4：为什么不等 Dream 自动处理记忆，而是自己写 memory skill？

**背景**：nanobot 的记忆管道设计是 "对话 → token 超限触发 Consolidator → 写 history.jsonl → Dream 每 2h 处理 → 更新 MEMORY.md"。

**问题发现**：
1. history.jsonl 始终为空（0 行），Dream 没有数据源
2. 根因：DeepSeek 对话短，不触发 Consolidator

**迭代过程（4 轮修改）**：

| 轮次 | 尝试 | 结果 |
|------|------|------|
| 1 | AGENTS.md 写"Update MEMORY.md after interaction" | Bot 口头说"我记住了"，文件未变 |
| 2 | 改 AGENTS.md 显式要求 `write_file` | 框架 identity 模板有"do not edit directly"，Bot 抵触 |
| 3 | 创建独立 memory skill，`always: true` 加载 | 生效，但发现单用户覆盖多用户 |
| 4 | 改为 `### 用户名 (channel:chat_id)` 多用户格式 | 收敛到稳定方案 |

**结论**：框架的自动化机制在特定条件下会失效（短对话不触发 Consolidator），理解底层原理后主动兜底比死等框架自动化更可靠。

---

### 决策 5：跨平台用户信息怎么隔离？

**问题**：钉钉用户问"我是谁"，Bot 返回了飞书用户（王小明 + 美团）。

**根因**：MEMORY.md 最初用 `### 用户名` 存储，Bot 运行时上下文有 `Channel: dingtalk` + `Chat ID: staff_xxx`，但文件里只有名字，匹配不上，Bot 随机返回了第一个用户。

**解决**：
```
修改前：
### 王小明
- **Company:** 美团

修改后：
### 王小明 (feishu:ou_acef3ec12cf71b6e945b0082b2d91daa)
- **Company:** 美团

### 李大牛 (dingtalk:lli4xmky2ddh5c64znec2hqnhq)
- **Company:** 字节跳动
```

Bot 读取记忆时用 `Channel + Chat ID` 组合键精准匹配，两个平台互不干扰。即使两人同名（都叫王小明），只要 channel:chat_id 不同就是不同用户。

## 五、面试常见追问准备

### Q：nanobot 和 LangChain/AutoGPT 有什么区别？

nanobot 是轻量 Agent 框架（核心 agent loop 约 500 行），定位类似 Claude Code 的本地部署版。对比 LangChain：nanobot 不做 chain 抽象，直接用 function calling 驱动工具调用，更简洁但可扩展性不如 LangChain 的生态。

### Q：你为什么选 DeepSeek？

成本低（约 OpenAI 的 1/10），中文能力强。代价是指令遵循度不如 Claude，在"先读记忆再回复"这类多步指令上偶尔跳步。后续生产环境会考虑切到更强的模型。

### Q：这个系统能上线吗？

当前阶段：原型可用，功能完整，适合内部团队使用。上线前需要：模型升级、增加自动化测试、allowFrom 白名单收紧、敏感信息脱敏、监控告警。

### Q：如果日活到 1000 人会怎样？

瓶颈在 DeepSeek API 并发限制和 SQLite 写入锁。需要引入消息队列（如 Redis）做请求缓冲，SQLite 可换 PostgreSQL。但当前单机架构对百人以内团队完全够用。
