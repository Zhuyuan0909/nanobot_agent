# nanobot 项目面试介绍

## 一句话概括

**nanobot 是一个个人 AI 助手框架，支持多平台接入、工具调用和长期记忆，核心架构围绕 ReAct Agent 模式设计。**

---

## 项目背景（为什么做 / 为什么研究）

- 市面上有很多 AI 聊天应用，但大多数不能**自主使用工具**（读文件、搜代码、执行命令）或**跨平台工作**
- nanobot 解决的是：让 LLM 不仅能聊天，还能**像人一样操作电脑**，并且可以在 Telegram / Discord / 微信等多平台统一使用
- 它还是一个很好的 Agent 架构学习项目——代码量适中、模块清晰、涵盖 Agent 开发的几乎所有核心概念

---

## 技术架构（口述版）

用三句话描述架构：

1. **最外层是 Channel 层**——负责对接 Telegram / Discord / Slack / 微信等各种聊天平台，所有消息统一转成内部格式，通过 MessageBus 传递
2. **中间是 AgentLoop 调度层**——接收消息后，组装上下文（用户信息 + 长期记忆 + 对话历史 + 技能提示词），然后进入 LLM ↔ Tool 的迭代循环
3. **底层是 Provider + Tool 层**——Provider 负责调用各种 LLM API（OpenAI / Anthropic / DeepSeek），Tool 提供文件读写、Shell 执行、网页搜索等能力

```
聊天平台 → MessageBus → AgentLoop → ContextBuilder → Runner(LLM ↔ Tool) → 回复
```

## 面试中的技术亮点

### 1. ReAct Agent 循环

> "这个项目实现的是 ReAct 模式的 Agent"

LLM 每轮可以输出两种结果：要么是最终回复，要么是 Tool Call。Runner 收到 Tool Call 后执行工具、把结果追加到上下文、再次调用 LLM，直到 LLM 不再调用工具为止。

**关键细节：**
- 最大迭代 200 轮防止死循环
- 空回复自动重试（最多 2 次）
- `finish_reason=length` 时自动追加 "continue" 继续生成
- 多个 Tool Call 并发执行（`asyncio.gather`）

### 2. 上下文窗口管理

> "上下文窗口有限，我做了多层治理策略"

| 策略 | 说明 |
|------|------|
| 工具结果裁剪 | 单结果超过 16000 字符自动截断 |
| Micro Compact | 重复工具调用折叠为占位符 |
| History Snip | 总 token 超限时从历史头部裁剪 |
| Consolidator | 用 LLM 把旧消息压缩成摘要存入 history.jsonl |

### 3. 三层记忆系统

> "记忆分短期、中期、长期三个层级"

- **短期**：Session JSONL 存储完整对话，下次对话直接注入上下文
- **中期**：Consolidator 在 token 压力大时用 LLM 把旧消息压缩成摘要
- **长期**：Dream 进程每 2 小时自动分析对话摘要，提取关键事实写入 MEMORY.md

面试官可能的追问：**"为什么不直接全部存下来？"**
→ 因为 LLM 的上下文窗口有 token 限制（如 64K），对话越长越贵还越慢，需要压缩和分层

### 4. 工具 Schema 校验 + 自修正

> "工具调用前有完整的参数校验，错误会回传给 LLM 让它自我修正"

```
LLM 输出 Tool Call → prepare_call 校验 → 类型转换 → JSON Schema 验证
  ├─ 通过 → 执行
  └─ 失败 → "Error: xxx. [Analyze the error and try a different approach.]" → LLM 修正
```

这是 Agent 鲁棒性的关键——LLM 生成的参数不一定合法，框架需要兜底。

### 5. 并发与状态隔离

> "全局用 Semaphore 限制并发数，同一 session 内部串行保证状态一致"

- 全局 Semaphore(3)：防止 LLM API 并发过高
- 同一 session 的消息排队处理，通过 pending_queue 实现 mid-turn injection
- Session 之间完全隔离，互不干扰

### 6. 多 Provider 抽象

> "支持 OpenAI / Anthropic / DeepSeek 等，统一接口 + 各 Provider 自定义重试策略"

- 基于抽象基类 `LLMProvider`，新增 Provider 只需实现 `chat_with_retry` 方法
- 内置指数退避重试，区分瞬态错误（429/5xx）和业务错误（quota 不足）

## 面试回答模板

### 面试官：介绍一下你做过的最有挑战性的项目

> "我深入研究过一个开源 AI Agent 框架叫 nanobot，它是一个完整的个人 AI 助手，支持在 Telegram、Discord 等多平台运行，能读文件、执行命令、搜索网页。
>
> 核心架构是 ReAct Agent 模式——LLM 可以在思考和执行工具之间循环迭代。我比较关注它的几个设计：一是上下文窗口管理，它用了多层治理策略来应对 token 限制；二是记忆系统分成了短期会话、中期摘要和长期事实三个层级；三是工具的参数学校验和错误自修正机制，保证 Agent 的鲁棒性。
>
> 我读了它的全部核心源码，写了详细的架构分析文档。这个过程让我对 Agent 框架的设计模式有了比较完整的理解。"

### 面试官：Agent 的 Loop 是怎么实现的？

> "核心在 AgentRunner.run() 这个方法里。它是一个 for 循环，每轮将 messages 发给 LLM，根据返回判断是 tool_calls 还是最终回复。有 tool call 就执行工具、追加结果，continue 下一轮；没有就 break 返回结果。
>
> 在发给 LLM 之前还有一步上下文治理——包括裁剪过长结果、压缩重复内容、超限时删历史头部。这些保证了上下文始终在窗口限制内。"

### 面试官：Agent 怎么记住之前说过的事？

> "它用了三层记忆。短期记忆就是完整的对话记录存在 Session JSONL 文件里，下次对话直接加载。中期记忆通过 Consolidator 实现——当上下文 token 超过阈值时，用 LLM 把旧消息压缩成摘要。长期记忆通过一个独立的 Dream 进程，每 2 小时自动分析摘要、提取关键事实写入 MEMORY.md 文件。"

## 面试高频追问速答

**Q: Agent 调用工具的格式是怎么约定的？**
> 使用 OpenAI 的 function calling 格式。每个 Tool 通过 `to_schema()` 生成 JSON Schema，注入到 system prompt 后的 tools 参数中。LLM 返回的 tool_calls 解析后由 ToolRegistry 分发执行。

**Q: 如果 LLM 调用了一个不存在的工具或传了错误参数怎么办？**
> prepare_call 会校验工具名是否存在、参数类型是否匹配、required 字段是否齐全。校验失败直接返回带 hint 的错误信息给 LLM，让它自行修正。

**Q: 上下文窗口超了怎么办？**
> 四层治理：工具结果截断 → 重复内容折叠 → 历史头部裁剪 → Consolidator LLM 压缩。确保永远不会超过窗口限制。

**Q: 怎么保证 Agent 不会执行危险命令？**
> 有 restrict_to_workspace 模式限制文件操作范围，Shell 执行支持 sandbox，URL 请求有 SSRF 白名单。工具层面做了权限边界。

**Q: 这个项目用了什么技术栈？**
> Python 3.10+，asyncio 异步架构，依赖 openai SDK 和 anthropic SDK 做 LLM 调用，JSONL 做会话持久化，Git 做记忆版本管理。
