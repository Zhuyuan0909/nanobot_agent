# 飞书用户提问 — Agent 完整处理流程

## 流程总览

当用户在飞书发送"我的数据同步失败了怎么办？"时，整个链路分为 **7 个阶段**，从 WebSocket 接收消息到流式回复用户。

---

                                          ▼                  ▼                  ▼

## 阶段 0：Gateway 启动初始化

`nanobot gateway` 启动时的初始化顺序：

```
config.json 加载
  → MessageBus 创建（异步双向消息队列）
  → LLM Provider 连接（DeepSeek OpenAI 兼容接口）
  → SessionManager 初始化（JSONL 文件读写）
  → CronService 启动（加载 cron/jobs.json）
  → AgentLoop 构建（核心处理引擎）
  → ChannelManager 初始化（飞书 + 钉钉通道）
  → HeartbeatService 启动
  → 所有组件 asyncio.gather 并发运行
```

### 与你二次开发相关的关键初始化

**ContextBuilder 加载 Bootstrap 文件：**
- `AGENTS.md` → Bot 身份：CloudSync 客服专家，知识库优先，禁止编造
- `SOUL.md` → 服务原则：共情、高效、透明
- `USER.md` → 用户画像：猫尼斯，美团，MySQL→TiDB 迁移中
- `TOOLS.md` → 工具使用提示

**Skill 系统扫描 `workspace/skills/*/SKILL.md`：**
- 3 个 skill 的 frontmatter 都设置了 `always: true`，全文注入 system prompt
- faq/SKILL.md → 教 LLM 如何搜索 knowledge_base/*.md
- ticket/SKILL.md → 教 LLM 5 种工单操作 + SLA 升级规则
- memory/SKILL.md → 教 LLM 如何用 edit_file 更新 MEMORY.md

**MCP 服务器连接：**
- 启动 `python workspace/mcp_servers/ticket_server.py` 子进程
- 通过 JSON-RPC 协议发现 5 个工具
- 包装为 `mcp_ticket_*` 注册到 ToolRegistry

## 阶段 2：接收用户消息

```
WebSocket 事件到达
  → 去重检查（LRU 1000 条 message_id 缓存）
  → 过滤机器人消息
  → 群聊 @提及检测（groupPolicy: "mention"）
  → 添加 👍 反应（表示处理中）
  → 解析消息内容（文本 / 富文本 / 图片等）
  → allowFrom 白名单校验
  → 构建 InboundMessage 对象
  → 发布到 MessageBus.inbound
```

InboundMessage 结构：
```python
InboundMessage(
    channel="feishu",
    sender_id="ou_acef3ec12cf71b6e945b0082b2d91daa",
    chat_id="ou_acef3ec12cf71b6e945b0082b2d91daa",
    content="我的数据同步失败了怎么办？",
    metadata={"_wants_stream": True}  # config.streaming = True
)
```

## 阶段 3：AgentLoop 核心处理

这是整个流程的**核心环节**，nanobot 框架在这里把你的自定义组件编排起来。

### 3.1 会话管理

- `unifiedSession: true` → session key = `"unified:default"`
- 飞书和钉钉共享同一会话历史
- 从 `workspace/sessions/unified_default.jsonl` 恢复历史对话
- AutoCompact 检查会话是否过期
- Consolidator 检查 token 是否超预算，超限时 LLM 摘要旧消息

### 3.2 构建 System Prompt（每次 LLM 调用前完整组装）

```
[Identity 模板]
  运行时信息（OS、Python 版本）、workspace 路径、格式化提示
    ↓ 拼接
[AGENTS.md]
  CloudSync 客服专家身份，知识库优先原则，禁止编造答案，
  5 步工作流定义，升级协议
    ↓ 拼接
[SOUL.md]
  服务原则：共情、高效、透明
    ↓ 拼接
[USER.md]
  用户画像：猫尼斯（美团），数据平台负责人，
  MySQL→TiDB 迁移中，偏好列表格式和可复制命令
    ↓ 拼接
[TOOLS.md]
  工具使用提示：grep vs exec 的选择，安全限制
    ↓ 拼接
[Memory]
  读取 memory/MEMORY.md，注入用户长期记忆
  当前包含："正在从 MySQL 迁移到 TiDB"
    ↓ 拼接
[3 个 Always Skill 全文注入]
  ├── faq/SKILL.md
  │    知识库搜索策略（grep + read_file）
  │    匹配策略（完全/部分/无匹配）
  │    规则：FAQ 未命中 = 立即创建工单
  ├── ticket/SKILL.md
  │    5 种 MCP 工单工具说明
  │    优先级定义（urgent/high/medium/low）
  │    升级 SLA（30min/4h/24h/72h）
  │    满意度回访流程
  │    回复模板
  └── memory/SKILL.md
      用 edit_file 更新 MEMORY.md
      记录什么 / 不记录什么
    ↓ 拼接
[Skills Summary]
  所有 skill 的摘要列表
    ↓ 拼接
[历史对话]
  session JSONL 中的近期消息轮次
```

### 3.3 LLM 推理循环（AgentRunner 控制）

LLM 收到完整的 system prompt + 用户问题 `[Runtime Context]\n\n我的数据同步失败了怎么办？` 后，按照 `AGENTS.md` 定义的 5 步工作流执行：

```
Step 1: 读取记忆
  → MEMORY.md 已注入 system prompt
  → LLM 已知用户在做 MySQL→TiDB 迁移

Step 2: 分类问题
  → "数据同步失败" → 归类为 troubleshooting 类型

Step 3: 路由决策
  → 调用 grep 搜索 knowledge_base/troubleshooting.md 中"同步失败"相关内容
  → 调用 read_file 读取匹配章节的完整内容
  
  三种结果分支：
  ├── 完全匹配 → 直接回答（综合知识库内容）
  ├── 部分匹配 → 分享已知信息 + 创建工单
  └── 无匹配   → 立即创建工单（不编造答案）

Step 4: 回复用户
  → 按 ticket/SKILL.md 定义的模板格式化回复
  → 流式输出（每段 delta 通过 MessageBus 发送到飞书）

Step 5: 更新记忆
  → 如有新的用户信息，调用 edit_file 更新 memory/MEMORY.md
```

### 3.4 MCP 工具调用链路（以创建工单为例）

```
LLM 决定调用 mcp_ticket_create_ticket(title="数据同步失败", ...)
  ↓
ToolRegistry 路由到 MCPToolWrapper
  ↓
JSON-RPC 请求写入 ticket_server.py 子进程的 stdin
  {
    "jsonrpc": "2.0",
    "method": "tools/call",
    "params": {
      "name": "create_ticket",
      "arguments": {
        "title": "数据同步失败",
        "description": "用户在数据迁移过程中遇到同步失败问题",
        "priority": "high",
        "user_id": "ou_acef3ec12cf71b6e945b0082b2d91daa"
      }
    },
    "id": 1
  }
  ↓
TicketMCPServer._create_ticket()
  → 生成工单号 TK-A3F8B2C1
  → INSERT INTO tickets (...)
  → 写入 workspace/data/tickets.db
  ↓
返回 JSON-RPC 响应
  {"jsonrpc": "2.0", "result": {"success": true, "ticket_id": "TK-A3F8B2C1"}, "id": 1}
  ↓
工具结果回传给 LLM
  ↓
LLM 将工单号编入自然语言回复
```

## 阶段 4：流式输出到飞书

```
LLM 生成每个 token delta
  → on_stream 回调发布 StreamDelta 到 MessageBus.outbound
  → ChannelManager._dispatch_outbound() 消费
  → FeishuChannel.send_delta() 调用飞书 CardKit 流式 API
  → 首次 delta：创建流式交互卡片
  → 后续 delta：更新卡片内容（打字机效果）
  → 流结束：关闭流式模式，移除 👍，添加 ✅ 反应
```

---

## 阶段 5：回复投递

ChannelManager 的 outbound dispatcher 一直在循环消费 MessageBus：

```python
# 非流式消息的投递逻辑
Consumer → 查表 self.channels["feishu"]
  → FeishuChannel.send()
  → 检测内容格式：
    ├── 复杂 Markdown（表格/代码块/标题）→ interactive 卡片
    ├── 中等文本含链接 → post 富文本
    └── 短纯文本 → text 纯文本
  → 调用飞书 API 发送
  → 处理媒体附件（图片/文件上传）
```

## 阶段 6：持久化与后台任务

本轮对话结束后：

| 操作 | 说明 |
|---|---|
| **Session 保存** | assistant 回复 + 工具调用结果写入 `unified_default.jsonl` |
| **Token Consolidation** | 如果 prompt 超 token 预算，后台 LLM 摘要旧轮次归档到 `history.jsonl` |
| **Dream 记忆维护** | 每 2 小时 cron 触发，从 `history.jsonl` 提炼关键事实，通过 edit_file 更新 `MEMORY.md` |
| **Heartbeat 检查** | 每 30 分钟检查 `HEARTBEAT.md`，如有活跃任务则执行 |

## 核心机制总结

**nanobot 框架的角色是编排引擎**——它不定义 Agent 的行为，而是把你的声明式配置组装成 system prompt + 工具列表，交给 LLM 去执行。

你二次开发的所有组件通过以下两个机制融入框架：

1. **System Prompt 注入**：`AGENTS.md`、`SOUL.md`、`USER.md`、`TOOLS.md`、`skills/*/SKILL.md`、`memory/MEMORY.md` 在每次 LLM 调用前完整组装进 system prompt
2. **MCP 工具注册**：`ticket_server.py` 作为独立子进程运行，工具通过 JSON-RPC 发现并注册到 ToolRegistry，LLM 可以像调用内置工具一样调用它们
